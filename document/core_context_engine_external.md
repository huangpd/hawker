# Hawker 核心上下文引擎

## 1. 问题

大多数 Agent 在任务变长后都会变重。

原因通常不是模型不会推理，而是上下文组织方式出了问题：

- 历史消息只增不减
- 失败尝试持续残留
- 页面结构反复被整页塞回 prompt
- 模型需要从旧代码里“猜”当前状态

对 Web Agent 而言，这个问题尤其严重，因为 DOM 往往是上下文中最昂贵的一部分。

Hawker 解决的核心问题，不是“如何再多压一点 token”，而是如何让模型始终工作在一个高密度、可持续更新的工作区里。

---

## 2. 核心思路

Hawker 将传统的线性对话流重构为状态流。

模型看到的重点不再是漫长聊天历史，而是一个持续更新的 `Notebook Workspace`。这个工作区只保留真正对下一步决策有用的信息：

- 当前进度
- 已确认的成功节点
- 已知失败经验
- 当前可复用变量
- 当前页面结构证据

这意味着模型不再被迫“阅读过去”，而是可以直接“面向当前状态”行动。

---

## 3. 为什么这是关键技术

这套技术的价值不只是省 token，更在于它把 Agent 从“历史驱动”变成“状态驱动”。

### 3.1 长短期记忆分离

Hawker 将上下文分成两层：

- 短期记忆：保留最近少量原始步骤，用于调试和局部修正
- 长期记忆：把更早历史提纯为里程碑和经验教训

这样模型既不会失去局部调试能力，也不会被早期无效尝试拖累。

### 3.2 变量不再依赖历史代码回忆

通过 `Namespace Snapshot`，模型可以直接看到当前有哪些持久化变量、变量类型和样本摘要。

这让 Agent 不需要反复回看过去定义过什么，减少了大量无意义 token 消耗。

### 3.3 DOM 被降级为可衰减证据

Hawker 并不把 DOM 当作默认长期上下文，而是将页面结构分成三种层级：

- `summary`
- `diff`
- `full`

其中完整 DOM 只允许短暂存在，随后自动折叠为轻量摘要。

这一步非常关键，因为它直接解决了 Web Agent 中最常见的上下文膨胀来源。

---

## 4. Hawker 的浏览器智能为什么不走 DOM-first

Hawker 的浏览器智能遵循一条明确原则：

**API-first，Browser-assisted，DOM-as-evidence。**

含义是：

- 优先发现和利用接口
- 浏览器主要负责建立真实会话、发现结构和执行交互
- DOM 是结构证据和兜底手段，而不是默认数据源

这样做有两个直接收益：

1. 数据获取路径更稳定
2. prompt 更轻

换句话说，Hawker 不是单纯“更会抓 DOM”，而是更早地把问题从页面层转成数据层。

---

## 5. 自动上下文调度

Hawker 不要求模型每次都显式决定“我要不要完整 DOM”，而是由系统自动调度上下文级别。

例如：

- 导航默认只给 `summary`
- 点击后优先给 `diff`
- 连续无进展时自动升级到 `full`
- 调试窗口结束后再自动折叠回摘要

这意味着系统能在“足够信息”和“可控成本”之间动态平衡，而不是永远保守或永远激进。

---

## 6. 可观测性不是附加功能

Hawker 的一个重要特点是，这套上下文系统不是黑盒。

系统会导出完整的 `llm_io.json`，记录：

- 每一步真实发送给模型的 prompt
- prompt 是如何拆分出来的
- 哪些内容被折叠或省略
- 模型原始返回
- 解析结果与执行结果

这使得上下文优化变成一个可以被量化、被分析、被持续迭代的工程问题，而不是“凭感觉调 prompt”。

---

## 7. 本质上，Hawker 在做什么

Hawker 的上下文引擎，本质上是在完成一次角色转换：

把大模型从一个需要阅读冗长历史的被动执行器，转变成一个面向工作区状态做决策的主动调度器。

它依赖的不是单一技巧，而是一组协同机制：

- 状态流替代聊天流
- 长短期记忆分离
- Namespace Snapshot
- DOM 分级与生命周期控制
- 自动上下文调度
- 可审计的 LLM I/O

这也是 Hawker 的核心技术壁垒所在。

---

## 8. 一句话总结

Hawker 不是简单地“压 prompt”，而是在为 Web Agent 构建一个可持续运行的上下文操作系统。
