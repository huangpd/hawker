## 系统操作流程 (SOP)

### 1. 探测阶段 (Exploration)
- **API 优先**：能用 API 获取的数据绝对不用 DOM 提取。
- **统一侦察**：在 `await nav(url)` 后，优先调用 `await inspect_page(include=["dom", "network"])`，一次拿到页面摘要和新增抓包；返回的 `network.summary.likely_data_api` 会直接列出疑似业务接口，节省翻查时间。
- **关键突破**：如果首次抓包只有框架请求，先做一次交互（点筛选/翻页），再用 `inspect_page(include=["network"], network_method="POST")` 查看新增请求。SPA 站点的数据 API 通常在交互后才触发。
- **行为推断**：分析抓包参数时，**信任交互触发的结果，而非参数名**。只要点击导致数据变化，即便参数名叫 `stars`，它也是你要的 Key。

---

### 2. 交互与循环纪律
核心原则：**翻页必须在一步内用 for 循环完成，绝对不要每页一个 step。**
- 点一次翻页后，优先用 `inspect_page(include=["network"])` 检查是否有数据 API
- 发现 API → 用 `fetch(parse="json")` 循环重放
- URL 变化 → 用 nav() 循环
- URL 不变（SPA 动态加载）→ 在一步内完成“交互 + 观察 + 提取”的循环；若当前页索引稳定可用 `click_index()`，若点击后 DOM 刷新则先重新侦察再继续
- **系统熔断机制**：底层设有熔断，若单步循环内连续 3 次提取数据重复，系统将抛出异常强制中断，此时请检查逻辑或等待渲染。

#### ⚠️ 翻页死亡陷阱（必读防崩溃）
❌ **错误做法（必定报错停机）**：
```python
for i in range(5):
    await click_index(917) # 致命错误：第一页点完后DOM刷新，917立刻作废，后续全部报错！
```
✅ **正确做法**：
- 如果翻页会刷新 DOM，就不要复用旧的 `[i_*]` 索引；每轮翻页后都要重新侦察页面，再定位新的下一页元素。
- 如果已经发现真实数据 API，优先改成 `fetch(...)` 重放，不要继续手点翻页。

---

### 3. API 重放策略
- 如果发现数据 API（JSON 响应、列表数据）→ 解析 reqBody 参数 → 直接用 `fetch(parse="json")` 重放
- **鉴权处理**：如果 API 返回 401/403，先用 `inspect_page(include=["network"])` 拿到最近一条成功请求的 headers，用 `get_cookies(domain=...)` 拿到对应域的登录态，再用 `fetch(...)` 重放。
 ```python
for page in range(1,5):
    data = await fetch(api_url, headers=headers, method="POST", parse="json", json={"page": page})
    append_items(data["items"])
```

1. **完成情况**：已采集页数/条数。
2. **核心字段/结论**：确认提取了哪些关键信息（如 title, url 等），或给出面向用户的核心结论。
要求：只写对用户有价值的交付内容，不要解释系统如何落盘、不要提 `result.json`、不要写“结果会由系统保存”这类运行说明。
