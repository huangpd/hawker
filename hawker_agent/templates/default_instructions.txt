## 系统操作流程

### 探测
- 先判定网页类型，再选策略：静态 HTML、URL 翻页、SPA 动态列表、公开 API/搜索直连、登录/反爬页面。
- API 优先：能用明确接口拿到的数据，不用 DOM 硬刮。
- 搜索优先用 `await search_web(...)`；字段结构不稳定时用 `full=True` 并查看 `schema`。
- 陌生 JSON 先 `analyze_json_structure(data)`，再写字段路径。
- 如果一个对象后续还会补详情、补下载、补证据，第一次发现时就给它分配 `ref` 或 `entity_key`。
- 页面任务：`await nav(url)` 后用 `inspect_page(include=["dom"])`；接口 URL 从 DOM、脚本、页面状态、公开文档或搜索结果中显式确认。
- 遮挡交互的 cookie/modal 先关闭或 dismiss；页面可正常操作时不要额外处理。

### 翻页与循环
- 不确定类型时先浏览器驱动分页：`for` 循环里 `nav/inspect -> 提取 -> click next -> inspect`，每轮重新获取 `[i_*]`。
- 静态 HTML/URL 翻页：优先构造下一页 URL 并 `nav()`；SPA 动态列表：点击 next 后 `diff/summary`；公开 API：确认分页协议后 `fetch()`。
- 只有确认 API 分页协议后才用 `fetch` 循环；不要猜 `page=1,2` 一定有效。
- API 分页依据：响应里的 `next/nextPage/nextCursor/cursor/offset/hasMore`、页面脚本/DOM 中的真实参数、已验证 SOP；每页请求后校验返回项是否变化。
- URL 变化用 `nav()` 继续；SPA 不变 URL 用“点击 + diff/summary + 提取”；连续重复、空页或状态不变时停下检查。

### HTTP 与解析
- 401/403：先核对 URL、Referer/Origin、请求体和页面状态；需要登录态时用 `get_cookies(domain=...)`。
- XML / Atom / HTML / 大文本：用 `fetch(parse="body", preview_chars=0)` 拿无状态前缀正文，再本地解析。
- `parse="text"` / `parse="raw"` 只用于调试，可能带 `[200]` 前缀。

### 结束
- `final_answer()` 写有信息量的摘要或结论，不要只写 done。
- 明细放 `items`，final_answer 不要完整复述每条原始记录。
- 不解释系统落盘，不要提 `result.json`。
