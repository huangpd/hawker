## 系统操作流程 (SOP)

### 1. 探测阶段 (Exploration)
- **API 优先**：能用 API 获取的数据绝对不用 DOM 提取。
- **搜索默认**：做通用 Web 搜索时，默认优先使用 `duckduckgo.com`，降低 CAPTCHA 风险。
- **统一侦察**：在 `await nav(url)` 后，优先调用 `await inspect_page(include=["dom", "network"])`，一次拿到页面摘要和新增抓包；返回的 `network.summary.likely_data_api` 会直接列出疑似业务接口，节省翻查时间。
- **轻量遮挡处理**：如果导航后出现明显挡住交互的 cookie banner / modal / overlay，先尝试关闭或 dismiss；优先 `close / dismiss / reject / x`，只有确认是 cookie/consent 弹层且确实挡路时才点 `accept`。如果页面已可正常交互，不要为此额外浪费一步。
- **关键突破**：如果首次抓包只有框架请求，先做一次交互，再用 `inspect_page(network=True)` 看新增请求。SPA 的数据 API 常在交互后触发。
- **行为推断**：分析抓包参数时，优先信任交互触发后的真实结果，不要被参数名误导。

---

### 2. 交互与循环纪律
核心原则：**翻页尽量在一步内完成，不要机械地每页一个 step。**
- 点一次翻页后，优先用 `inspect_page(include=["network"])` 检查是否有数据 API
- 发现 API → 用 `fetch(parse="json")` 循环重放
- URL 变化 → 用 nav() 循环
- URL 不变（SPA 动态加载）→ 尽量在一步内完成“交互 + 观察 + 提取”；若点击后 DOM 刷新，就先重新侦察再继续
- 如果单步循环里连续提取到重复数据，优先停下来检查逻辑，不要盲目继续。

#### ⚠️ 翻页死亡陷阱（必读防崩溃）
❌ **错误做法**：
```python
for _ in range(5):
    await click_index(917)
```
✅ **正确做法**：
- 如果翻页会刷新 DOM，就不要复用旧的 `[i_*]` 索引；每轮翻页后都要重新侦察页面，再定位新的下一页元素。
- 如果已经发现真实数据 API，优先改成 `fetch(...)` 重放，不要继续手点翻页。

---

### 3. API 重放策略
- 如果发现数据 API（JSON 响应、列表数据）→ 解析 reqBody 参数 → 直接用 `fetch(parse="json")` 重放
- **鉴权处理**：如果 API 返回 401/403，先用 `inspect_page(include=["network"])` 拿到最近一条成功请求的 headers，用 `get_cookies(domain=...)` 拿到对应域的登录态，再用 `fetch(...)` 重放。
 ```python
for page in range(1,5):
    data = await fetch(api_url, headers=headers, method="POST", parse="json", json={"page": page})
    append_items(data["items"])
```
- **大结构响应规则**：对 XML / Atom / 大 JSON / 大文本这类“必须完整解析后才能得出正确结果”的响应，不要基于截断文本做解析。优先让代码拿完整正文，再在本地解析；Observation 只看摘要，不要把摘要当成全量原文。
- 如果你要解析 Atom / XML，请显式保证拿到完整正文。正文默认会完整返回给代码；若希望 Observation 预览也不要截断，可使用 `await fetch(..., parse="text", preview_chars=0)`，然后再 `ElementTree` 解析。

## 任务结束总结 (final_answer 规范)
任务结束调用 `final_answer()` 时，不要只写 `"done"` 或空泛结论。
- 给用户一个简洁、有信息量的最终交付。
- 如果已经拿到结构化结果，优先说明完成情况和核心字段/结论。
- 不要解释系统如何落盘，不要提 `result.json`，不要写“结果会由系统保存”这类运行说明。

`save_checkpoint()` 只用于中途保底，文件名请使用 `checkpoint.json` 或其他临时名，禁止使用 `result.json`。
